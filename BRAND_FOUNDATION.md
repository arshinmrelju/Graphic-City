# GraphicCity — Brand Foundation

---

## Brand Story

GraphicCity was born from a quiet frustration. Every tool, every template, every "creative solution" on the market promised blazing speed and zero effort — and delivered forgettable work. The industry had confused efficiency with excellence.

We started GraphicCity to restore what had been lost: the belief that design is not a commodity. It is a craft.

We are a creative design studio that treats every pixel, every transition, every micro-moment as an opportunity to communicate. We do not mass-produce. We make. We do not optimize for speed at the cost of soul. We optimize for feeling, for clarity, for permanence.

GraphicCity exists for founders, leaders, and teams who refuse to blend in. Who understand that great design is not a coat of paint — it is the structure of thought itself. We partner with people who care enough to demand more from every surface their brand touches.

We are not a factory. We are a studio. And every project is a statement.

## Typography System

### Primary Typeface — Editorial

**ABC Social** (or **Sharp Grotesk** / **TWK Everett** as alternatives)

- A humanist grotesk — structured but warm, technical but approachable.
- Used for: headlines, hero text, large display typography, navigation.

### Secondary Typeface — UI & Body

**Inter** (or **Suisse Int'l**)

- Highly legible at every size. Perfect for long-form reading, UI labels, captions.
- Used for: body copy, UI elements, forms, tables, specifications.

### Monospace — Code & Data

**JetBrains Mono** (or **SF Mono**)

- Used for: code blocks, data display, technical specifications, legal fine print.

|---|---|---|

### Letter-spacing

| Usage | Tracking |
|---|---|
| Display & H1 | -0.02em |
| H2–H4 | -0.01em |
| Body | 0em |
| Caption & Fine | +0.03em |

### Philosophy

Typography is the voice of the interface. We prioritize readability, rhythm, and hierarchy over decorative flourishes. Type weight is used to create contrast, not size. We prefer fewer sizes with more deliberate spacing over a sprawling scale.

## 8px Spacing System

Every spacing value is a multiple of 8px. This creates a predictable, harmonious vertical and horizontal rhythm.

| Token | Value | Usage |
|---|---|---|
| `space-1` | 4px | Tightest — icons inside buttons, micro adjustments |
| `space-2` | 8px | Tight — between related elements, icon gaps |
| `space-3` | 12px | 8*1.5 callout — fine-tuned internal padding |
| `space-4` | 16px | Base — component padding, content gaps |
| `space-5` | 24px | Comfortable — section spacing, card padding |
| `space-6` | 32px | Wide — between sections, modal padding |
| `space-7` | 48px | Section — major layout breaks |
| `space-8` | 64px | Page — between page sections |
| `space-9` | 96px | Generous — hero bottom padding, large containers |
| `space-10` | 128px | Maximum — page edge padding, large section gaps |

Exceptions to the 8px grid are permitted only in typography line-height and icon internal spacing — and only when the optical adjustment improves perceived balance.

## Border Radius System

| Token | Value | Usage |
|---|---|---|
| `radius-none` | 0px | Sharp edges — editorial containers, full-bleed sections |
| `radius-sm` | 4px | Subtle — input fields, small components |
| `radius-md` | 8px | Standard — cards, modals, buttons, panels |
| `radius-lg` | 12px | Generous — large cards, dialog boxes |
| `radius-xl` | 16px | Extra generous — sheets, drawers |
| `radius-full` | 9999px | Pill — tags, badges, avatars |

### Philosophy

Radii are intentional. We prefer sharpness over softness by default — it communicates precision and confidence. Curves are added only when humanizing a component (buttons, inputs) or creating containment (cards, modals). We never radius-shape a container beyond 16px in standard layout. Pill shapes are reserved exclusively for small elements.

| `shadow-lg` | `0 8px 24px rgba(0,0,0,0.08)` | Modals, sheets, large dialogs |
| `shadow-xl` | `0 16px 48px rgba(0,0,0,0.10)` | Overlays, full-screen menus |
| `shadow-soft` | `0 20px 60px rgba(0,0,0,0.06)` | Premium — featured content, hero sections |

### Philosophy

Shadows are subtle. We never use hard, dark shadows. Our shadows are soft, wide, and barely perceptible — they suggest depth without announcing it. The goal is to create a gentle hierarchy where surfaces feel stacked, not floating.

## Icon Style

- **Weight:** 1.5px stroke (medium-thin)
- **Style:** Outlined, geometric, with 2px rounded caps and rounded joins
- **Size grid:** 24×24px base, rendered on a 2px internal grid
- **Filled variants:** Never used in UI. Reserved for status states only (checked, active).
- **Philosophy:** Icons are functional, not decorative. They communicate an action or object instantly. If an icon requires explanation, replace it with text. Consistency of stroke weight and corner radius across the full set is non-negotiable.

---

## Illustration Style

- **Approach:** Abstract geometric figuration. Think simplified, architectural forms — not literal, not cartoonish.
- **Color:** Monochromatic or duotone (Core Black + Signal Blue, or Stone neutrals).
- **Line work:** Consistent 2px stroke, matching icon system.
- **Fill:** Minimal. Negative space carries the form.
- **Tone:** Sophisticated, slightly surreal. Should feel more like a diagram of an idea than a literal drawing.
- **Rule:** Never use illustration to explain the obvious. Use it to make the abstract feel tangible.
- **AI policy:** Illustration is hand-crafted or directed by human hands. No generative filler.

---

## Photography Direction

- **Style:** Editorial, cinematic, documentary-natural
- **Lighting:** Natural or motivated — no harsh on-camera flash. Deep shadows, open highlights.
- **Color grading:** Desaturated with selective warmth. Never oversaturated. Think Wong Kar-wai meets Peter Lindbergh.
- **Subjects:** Real people in real spaces. Candid, not posed. Eyes engaged, not performing.
- **Composition:** Breathing room. Generous negative space. The subject does not need to fill the frame.
- **Post-processing:** Film-emulation grain (subtle), gentle contrast curves. No HDR. No clipping.
- **Rule:** Every photograph should feel like a still from a film you would want to watch.

## UI Philosophy

**1. Content is the interface.**
The UI should fade so the work can speak. Every ornament that does not serve clarity is removed.

**2. Hierarchy through weight, not size alone.**
Before increasing font size, try increasing weight. Before adding color, try adding space. The most elegant hierarchy is invisible.

**3. Generous whitespace is a feature.**
Cramped interfaces signal insecurity. We leave room for the eye to rest. Space is not wasted — it is structure.

**4. One primary action per view.**
Every screen should answer one question. Every button, link, and label should support that question. If a view has two primary actions, the view is wrong.

**5. Consistency is not uniformity.**
Patterns repeat, but every surface is considered in context. A hero section and a settings panel do not need the same spacing, but they should feel like they belong to the same system.

**6. Dark mode is not an afterthought.**
Dark surfaces are never pure black. They are deep, warm grays (Core Ink, Stone 900). Text is never pure white on dark — it is soft white (`#F5F5F5`). Contrast should be sufficient for legibility but gentle enough to read for hours.

**7. Inputs should feel tactile.**
Even in a flat design system, form fields, buttons, and interactive elements should communicate affordance through subtle elevation changes, not borders alone.

---

## Motion Philosophy

**1. Motion has purpose.**
If a transition does not communicate hierarchy, spatial relationship, or state change, remove it. Motion is a functional tool, not decoration.

**2. Duration & Easing**

| Action | Duration | Easing |
|---|---|---|
| Micro-interactions (hover, toggle) | 150ms | ease-out |
| Moderate (panel open, scroll reveal) | 300ms | ease-out |
| Expressive (page transition, hero enter) | 500–700ms | ease-in-out custom |
| Ambient (slow parallax, subtle float) | 2000ms+ | linear or custom |

**3. The perfect easing curve.**
Default CSS easing is never used. Every curve is authored. Our signature curve:

```
cubic-bezier(0.25, 0.01, 0.15, 1)
```

Fast start, long tail, graceful finish. Nothing feels mechanical.

**4. Stagger, not simultaneous.**
When multiple elements animate, they enter in sequence — not at once. The delay between each element should be 50–80ms, creating a wave-like reveal.

**5. Exit matters.**
Elements exiting the screen deserve the same care as elements entering. Never abrupt disappearances. Fade + scale to 0.95 at minimum.

**6. Reduced motion is not an afterthought.**
A `prefers-reduced-motion` compliant version of every animation must exist. Replace transitions with opacity fades and instant state changes.

---

## Component Philosophy

**1. One component, one job.**
Every component does exactly one thing and does it completely. If a component is responsible for layout, state, and visual styling simultaneously, it is too large. Break it down.

**2. Composability over configuration.**
Prefer small, combinable primitives over monolithic components with 40 props. A Button should not also handle dropdowns, loading states, and tooltips. Compose those behaviors from separate components.

**3. Props are a public API.**
Every prop must be intentional. Avoid boolean props that toggle visual variants — prefer semantic variant props (`variant="primary"` / `variant="ghost"`). Boolean props should only exist for binary behavior, not visual modes.

**4. Defaults are opinions.**
Every component ships with opinionated defaults. A Button defaults to `variant="primary"`, a Dialog defaults to `size="md"`. These defaults encode our design decisions and prevent decision fatigue.

**5. Responsive is structural, not additive.**
A component is not "desktop with mobile overrides." It is responsive from the ground up. Mobile-first. The smallest screen receives the most constrained layout. Larger screens receive expansion, never patching.

**6. Accessibility is not a feature.**
It is not a prop. It is not a variant. Every component must be keyboard-navigable, screen-reader-accessible, and meet WCAG 2.1 AA at minimum before any visual work is considered complete.
